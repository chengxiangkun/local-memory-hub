# Outcome Roadmap：Local Memory Hub V1

## 1. 路线图目标

这份路线图把 Local Memory Hub V1 的功能清单，转换成按用户结果和业务验证驱动的阶段计划。

当前产品范围已经包含：

- 本地源资料库
- 快捷导入
- 外部文档导入
- 本地解析
- 大模型兜底闭环
- 搜索和问答
- 向量与图谱
- 污染治理
- 模型供应商系统
- MCP/API 外部调用

因此 V1 不能再按“想做什么功能就做什么功能”的方式推进。它需要按结果分阶段交付。

## 2. 总体结果目标

V1 要实现的总体结果：

> 让高信息密度用户可以低成本收集资料，清楚知道资料是否已被解析和进入记忆，并能通过搜索、问答、图谱和外部 AI 工具可信地重新使用这些资料。

## 3. 阶段路线图

## 阶段 1：让用户可靠保存和追溯源资料

### Outcome

让用户把文件、链接、剪贴板内容、截图等资料放进系统后，可以清楚知道资料已经被保存在哪里、来自哪里、当前是什么状态。

### 用户结果

用户不再担心资料丢失，也不再需要猜测导入是否成功。

### 主要输出

- Inbox
- 源资料库
- SourceRecord
- 本地文件存储
- 文件夹、日期、类型、来源排序
- 导入状态
- 源文件搜索
- 基础删除和回收站

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 支持来源保存成功率 | >= 90% |
| 源资料状态可见率 | 100% |
| 用户能找到刚导入资料的比例 | >= 90% |
| 导入失败原因可解释率 | >= 90% |

### 关键假设

- 用户需要一个本地源资料库，而不只是 AI 问答。
- 文件夹、日期、类型、来源、状态足以支撑早期管理。
- 源资料和记忆分层不会让用户困惑。

### 不做

- 深度图谱
- 完整问答
- 外部平台复杂同步
- 全自动微信读取

## 阶段 2：让用户低成本导入常用来源

### Outcome

让用户可以通过自然习惯把资料送进系统，而不是打开复杂表单手动整理。

### 用户结果

用户可以用拖拽、粘贴、快捷键、截图、链接和外部文档入口完成导入。

### 主要输出

- ImportRequest
- ImporterRegistry
- Import Pipeline
- FileImporter
- UrlImporter
- ClipboardImporter
- ScreenshotImporter
- FeishuImporter
- YoudaoImporter
- McpImporter
- 导入失败兜底
- 去重与合并

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 快捷导入完成时间 | <= 5 秒完成保存 |
| 重复导入识别率 | >= 80% |
| 外部文档导入路径可用 | 飞书、有道云至少有一种可用路径 |
| 用户首日导入资料数 | >= 5 条 |

### 关键假设

- 快捷导入能显著降低用户录入成本。
- 飞书和有道云是 V1 值得支持的外部来源。
- 外部平台 API 不稳定时，导出/上传兜底可以被用户接受。

### 不做

- 完美覆盖所有视频平台
- 所有外部平台实时同步
- 自动读取私密聊天记录

## 阶段 3：让资料变成可用记忆

### Outcome

让用户导入的资料可以被本地解析、清洗、分段、入向量，并明确显示是否已经进入 AI 记忆。

### 用户结果

用户知道哪些资料已经能被 AI 使用，哪些资料解析失败，为什么失败。

### 主要输出

- 本地解析器
- 解析状态机
- 解析失败原因
- chunk
- embedding
- 入记忆状态
- 解析结果追溯源文件
- 重新解析

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 常见文件解析成功率 | >= 85% |
| 入记忆状态可见率 | 100% |
| 解析失败原因可解释率 | >= 90% |
| 重新解析成功率 | >= 70% |

### 关键假设

- 用户关心“是否已入记忆”。
- 解析失败原因透明会提升信任。
- 本地解析质量足以支撑早期问答和搜索。

### 不做

- 全量视频视觉理解
- 高复杂度知识图谱推理
- 大规模多用户处理

## 阶段 4：让本地解析能力通过大模型兜底逐步变强

### Outcome

让本地解析失败时可以调用用户配置的大模型修复，并把修复结果沉淀成下一次可复用的本地规则或样例。

### 用户结果

用户不需要在本地解析失败后手动处理。系统会在必要时调用大模型，并逐步减少未来同类问题的 token 消耗。

### 主要输出

- ModelProviderAdapter
- ModelRouter
- Provider 设置页
- 自定义 OpenAI-Compatible Endpoint
- 自定义 Anthropic-Compatible Endpoint
- 本地模型接入
- LLM parse fallback
- parser improvement artifacts
- token 模式：省 token / 平衡 / 深度
- 调用日志

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 本地解析失败后的兜底成功率 | >= 70% |
| 至少沉淀解析规则类型 | >= 3 类 |
| 同类失败重复调用大模型次数下降 | 下降 30% |
| 模型调用日志完整率 | 100% |

### 关键假设

- 大模型结果可以转化为可复用的本地解析规则。
- 用户愿意配置自己的模型 API Key。
- 支持主流模型和自定义 endpoint 能降低长期维护风险。

### 不做

- 自训练大模型
- 完整模型市场
- 自动选择灰色代理或非可信模型服务

## 阶段 5：让用户不配置大模型也能搜索，配置后可以问答

### Outcome

让用户无论是否配置大模型，都可以找到资料；配置大模型后，可以基于本地记忆生成带引用的答案。

### 用户结果

用户不被模型配置卡住。没有模型时能搜索，有模型时能问答。

### 主要输出

- 全局搜索
- 源资料搜索
- 全文搜索
- 向量搜索
- 搜索结果融合
- Ask 页面
- 无大模型搜索兜底
- 大模型问答
- 引用来源
- 标记错误引用/污染引用

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 无模型状态下搜索可用率 | 100% |
| 问答答案引用覆盖率 | 100% |
| 用户点击引用回源资料比例 | >= 40% |
| 用户标记有用答案比例 | >= 60% |

### 关键假设

- 搜索兜底能让未配置模型用户继续使用。
- 引用来源是 AI 问答信任的关键。
- 用户愿意标记错误引用来改善系统。

### 不做

- 无来源答案
- 默认把污染资料纳入问答
- 没配置模型时假装生成总结

## 阶段 6：让用户治理污染记忆

### Outcome

让用户可以发现、隔离、删除或恢复污染数据，并控制源文件删除时是否删除解析文本、向量和图谱。

### 用户结果

用户可以撤回不想让 AI 使用的内容，避免错误资料持续影响搜索、问答和图谱。

### 主要输出

- 污染状态
- 隔离区
- 恢复
- 从向量中移除
- 从图谱中移除
- 删除派生产物
- 源文件删除高级选项
- 删除记录
- 影响范围查看

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 污染内容排除时间 | <= 5 秒 |
| 删除操作可理解率 | >= 80% |
| 污染内容进入普通搜索比例 | 0% |
| 删除后残留索引错误 | 0 个阻塞级问题 |

### 关键假设

- 记忆污染治理是用户信任的必要功能。
- 删除源文件和删除派生记忆的关系可以被用户理解。
- 隔离比直接删除更适合作为默认操作。

### 不做

- 自动删除用户未确认的高风险资料
- 不可恢复的默认硬删除

## 阶段 7：让用户通过图谱探索关系并形成传播点

### Outcome

让用户可以通过图谱理解资料、主题、想法、人物和项目之间的关系，并获得视觉上愿意分享的体验。

### 用户结果

用户不只是搜索单条资料，还能看到自己的信息如何互相关联。

### 主要输出

- 图谱页面
- 图谱搜索
- 节点聚类
- 节点详情
- 关系来源解释
- 时间线模式
- 污染节点过滤
- 图谱节点/边删除
- 图谱快照导出

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 图谱节点点击率 | >= 50% 的图谱访问会点击节点 |
| 图谱搜索成功定位率 | >= 80% |
| 用户从图谱回源资料比例 | >= 30% |
| 图谱快照分享意愿 | 内测用户中 >= 30% 表示愿意分享 |

### 关键假设

- 图谱既能有用，也能有传播感。
- 用户愿意用图谱探索资料关系。
- 图谱关系来源透明可以降低误导风险。

### 不做

- 完整社交网络
- 用户间关注、评论、私信
- 不可解释的 AI 关系

## 阶段 8：让外部 AI 工具安全调用本地记忆

### Outcome

让 Codex、Claude Desktop、Cursor 或其他 AI 工具可以在用户授权下调用本地记忆、源资料和图谱上下文。

### 用户结果

用户不需要反复复制粘贴上下文。外部 AI 工具可以按需读取本地记忆，并保持权限和来源可控。

### 主要输出

- 本地 API
- MCP Server
- memory.search
- memory.ask
- memory.get_context
- source.import
- source.search
- source.status
- graph.search
- graph.get_neighbors
- graph.trace_source
- 外部接入说明

### 成功指标

| 指标 | 目标 |
| --- | --- |
| Codex 接入可完成率 | >= 80% 内测用户可按文档完成 |
| API/MCP 调用可追溯率 | 100% |
| 外部调用权限错误 | 0 个阻塞级问题 |
| 开发者继续使用意愿 | >= 60% |

### 关键假设

- MCP/API 是开发者早期用户的强价值点。
- 外部 AI 调用本地记忆会比手动粘贴上下文更高效。
- 用户愿意配置和授权外部工具。

### 不做

- 默认开放远程访问
- 未授权读取本地记忆
- 没有日志的外部调用

## 4. 阶段依赖关系

```text
阶段 1：源资料和状态
  ↓
阶段 2：低成本导入
  ↓
阶段 3：解析和入记忆
  ↓
阶段 4：大模型兜底闭环
  ↓
阶段 5：搜索和问答
  ↓
阶段 6：污染治理
  ↓
阶段 7：图谱探索
  ↓
阶段 8：外部 AI 调用
```

其中阶段 6 的污染治理不应该太晚才做。最小污染隔离能力应从阶段 3 开始进入底层状态机，完整治理页面可以在阶段 6 补齐。

## 5. V1 范围控制建议

必须守住的 V1 核心：

- 源资料状态透明。
- 导入入口统一。
- 解析和入记忆可追溯。
- 无模型可搜索。
- 有模型可问答。
- 污染可隔离。
- 图谱可搜索。
- MCP/API 可接入。

可以后置的内容：

- 完整社交网络。
- 全平台视频解析。
- 全自动微信采集。
- 云同步。
- 团队协作。
- 复杂图谱编辑器。
- 自训练模型。

## 6. 近期最小验证顺序

### 验证 1：源资料状态是否足够清楚

用静态原型测试：

- 用户能否找到刚导入的文件。
- 用户能否判断是否解析成功。
- 用户能否判断是否已入记忆。
- 用户能否找到源文件。

### 验证 2：快捷导入是否明显降低负担

做可用 demo：

- 粘贴链接。
- 拖拽文件。
- 快捷键保存剪贴板。
- 截图导入。

### 验证 3：无模型搜索兜底是否可接受

测试：

- 未配置模型时搜索资料。
- 配置模型后生成答案。
- 用户是否理解两者差异。

### 验证 4：图谱是否真的有用

用真实资料生成小图谱：

- 50-100 个节点。
- 支持搜索定位。
- 支持点回源文件。
- 支持污染节点隐藏。

### 验证 5：Codex/MCP 是否是强价值点

让开发者用户完成：

- 导入项目资料。
- Codex 通过 MCP 查询本地记忆。
- Codex 回答时引用本地资料。

## 7. 路线图总结

Local Memory Hub V1 的路线不应该从“图谱很酷”或“支持很多平台”开始。

正确顺序是：

```text
先让资料可信地进入系统
再让资料变成可用记忆
再让用户能搜索和问答
再让用户能治理污染
再让图谱成为探索和传播点
最后让外部 AI 工具安全调用
```

如果这个顺序守住，产品会越来越稳。如果一开始就追求全平台导入、复杂图谱和深度 Agent，系统很容易变成一艘挂满霓虹灯但还没装方向盘的飞船。
