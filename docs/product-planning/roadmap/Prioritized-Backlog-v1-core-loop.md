# 优先级 Backlog：Local Memory Hub V1 核心闭环

## 1. 目标

本文件对 V1 核心闭环用户故事进行优先级排序。

用户确认 V1 的 6 个核心卖点全部必须覆盖：

1. 资料可信地进入系统。
2. 资料变成可用记忆。
3. 用户能搜索和问答。
4. 用户能治理污染。
5. 图谱成为探索和传播点。
6. 外部 AI 工具安全调用。

因此本文件不再讨论“要不要做”，只讨论：

- 先做什么。
- 后做什么。
- 哪些能力依赖底层地基。
- 哪些功能可以先做最小闭环，再逐步增强。

## 2. 优先级方法

本次使用两层方法：

### 2.1 MoSCoW

- Must：V1 必须有，否则核心闭环不成立。
- Should：V1 应该有，影响体验完整度。
- Could：有价值，但可后置。
- Won't：V1 不做。

### 2.2 RICE 简化评分

评分字段：

- Reach：影响用户范围，1-5。
- Impact：对核心价值影响，1-5。
- Confidence：当前信心，1-5。
- Effort：开发复杂度，1-5，越高越难。

简化得分：

```text
Score = Reach × Impact × Confidence / Effort
```

得分不是绝对真理。依赖关系优先于分数。

## 4. P0 执行顺序

## Milestone 0：工程地基

这部分不直接面向用户，但必须先做。

| 顺序 | 能力 | 原因 |
| --- | --- | --- |
| 0.1 | SourceRecord 数据模型 | 所有导入、追溯、状态都依赖它 |
| 0.2 | 状态机 | 导入、解析、入记忆、污染、删除都依赖它 |
| 0.3 | 本地文件存储 | 源资料必须先安全落地 |
| 0.4 | ImportRequest 标准模型 | 所有入口必须统一 |
| 0.5 | Import Pipeline | 避免后期导入系统变成屎山 |
| 0.6 | 任务队列 / Job | 导入和解析必须解耦 |

## 5. 总体开发顺序

推荐顺序：

```text
0. 工程地基
1. 源资料和状态
2. 导入入口
3. 本地解析和入记忆
4. 模型供应商和大模型兜底
5. 无模型搜索
6. 有模型问答和引用
7. 污染治理
8. 基础图谱和图谱搜索
9. MCP/API 外部调用
10. 图谱视觉和分享打磨
```

## 6. 第一轮开发建议

第一轮不要一口气做完整 V1。

建议第一轮目标：

> 用户可以导入本地文件和链接，看到源资料状态，完成本地解析，进入记忆，并在无大模型情况下搜索到资料。

第一轮故事：

- US-1.1 导入本地文件
- US-1.2 粘贴链接导入
- US-1.7 源资料状态可见
- US-2.1 本地解析常见文件
- US-2.4 内容切分并进入向量记忆
- US-2.5 记忆可追溯回源文件
- US-3.1 无大模型时搜索可用

为什么先做这个：

- 它覆盖“资料可信进入系统”和“资料变成可用记忆”的最短闭环。
- 不依赖复杂外部平台。
- 不依赖大模型配置。
- 可以最快验证用户是否需要这个产品。

## 11. 关键提醒

这 6 个核心卖点都必须有，但不能同时从 UI、图谱、MCP、外部平台、大模型兜底五条线一起开工。

最稳的路径是：

```text
先让资料进入系统可信
再让资料变成记忆可信
再让搜索和问答可信
再让治理可信
再让图谱好用好看
最后开放给外部 AI
```

这不是降低产品野心，而是给野心装骨架。
