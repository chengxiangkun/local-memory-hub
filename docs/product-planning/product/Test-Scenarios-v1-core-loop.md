# 测试场景：Local Memory Hub V1 核心闭环

## 1. 目标

本文件定义 Local Memory Hub V1 的端到端验收场景。

这些测试场景用于验证 6 个核心卖点：

1. 资料可信地进入系统。
2. 资料变成可用记忆。
3. 用户能搜索和问答。
4. 用户能治理污染。
5. 图谱成为探索和传播点。
6. 外部 AI 工具安全调用。

这些场景也会指导后续信息架构、低保真原型和 UI 设计。

## 2. 测试角色

| 角色 | 说明 |
| --- | --- |
| 普通知识工作者 | 导入和搜索资料 |
| AI 重度用户 | 配置模型并使用问答 |
| 治理型用户 | 处理污染、删除和恢复 |
| 开发者用户 | 通过 MCP/API 调用本地记忆 |

## 3. 场景 1：文件导入到无模型搜索

### 测试目标

验证用户在未配置大模型的情况下，也可以导入源文件、看到状态、本地解析、入记忆，并通过搜索找到资料。

### 起始条件

- 用户已安装并启动本地应用。
- 用户未配置外部大模型 API Key。
- 用户有一个可解析的 PDF 或 Markdown 文件。
- 本地文件存储和数据库可用。

### 用户角色

普通知识工作者。

### 测试步骤

1. 用户打开工作台。
   - 预期：页面显示导入入口、最近资料和搜索框。
2. 用户拖拽一个 PDF 到应用。
   - 预期：系统创建导入任务，并显示“已保存，等待解析”。
3. 用户进入源资料库。
   - 预期：列表出现该 PDF，显示文件名、类型、日期、来源、本地路径、解析状态、入记忆状态。
4. 系统执行本地解析。
   - 预期：解析状态从“等待解析”变为“解析中”，再变为“解析成功”。
5. 系统切分内容并生成记忆。
   - 预期：入记忆状态从“等待入记忆”变为“已入记忆”。
6. 用户在全局搜索框输入 PDF 中的关键词。
   - 预期：搜索结果返回源资料和命中片段。
7. 用户点击搜索结果。
   - 预期：打开源资料详情，并显示命中位置。

### 预期结果

- 文件导入成功。
- 源资料记录可见。
- 解析状态和入记忆状态可见。
- 未配置大模型时，搜索仍然可用。
- 搜索结果可以追溯到源文件。

### 边界场景

- 文件重复导入时，系统提示重复并支持合并。
- 文件解析失败时，源资料仍保留，并显示失败原因。
- 文件过大时，系统提示限制或进入后台处理。

## 4. 场景 2：链接导入到源资料追溯

### 测试目标

验证用户粘贴网页或视频链接后，系统可以保存链接、识别来源、创建源资料记录，并在无法解析时给出兜底提示。

### 起始条件

- 应用已启动。
- 用户未配置平台授权。
- 用户准备一个普通网页链接和一个视频平台链接。

### 用户角色

普通知识工作者。

### 测试步骤

1. 用户在工作台粘贴普通网页链接。
   - 预期：系统识别为 URL 导入。
2. 系统保存原始 URL 和 canonical URL。
   - 预期：源资料库出现该网页记录。
3. 用户粘贴视频平台链接。
   - 预期：系统识别来源平台，如 B 站、YouTube 或通用视频链接。
4. 如果无法直接解析视频内容。
   - 预期：系统保存链接，并提示用户可上传视频文件或等待后续解析。
5. 用户按来源平台筛选。
   - 预期：可以看到对应网页和视频链接。

### 预期结果

- 链接可以可信进入系统。
- 即使无法解析，源资料也不会丢失。
- 用户知道下一步该做什么。

### 边界场景

- 短链接应展开或保存原始短链接。
- 失效链接应显示“源不可访问”。
- 重复链接应合并到已有源资料。

## 5. 场景 3：飞书/有道导入兜底

### 测试目标

验证飞书和有道云导入路径在 V1 中可用，即使 API 或权限受限，也能通过导出、上传或复制内容完成导入。

### 起始条件

- 用户有一个飞书文档链接。
- 用户有一个有道云笔记链接或导出文件。
- 用户未必完成平台授权。

### 用户角色

普通知识工作者。

### 测试步骤

1. 用户选择“导入外部文档”。
   - 预期：页面显示飞书、有道云和通用文档入口。
2. 用户粘贴飞书文档链接。
   - 预期：系统识别为飞书来源。
3. 如果未授权。
   - 预期：系统保存链接，并提示授权、导出或上传文件。
4. 用户上传飞书导出文件。
   - 预期：系统将导出文件合并到同一个源资料或 source group。
5. 用户导入有道云笔记。
   - 预期：系统支持链接、导出文件或复制内容导入。
6. 用户查看源资料详情。
   - 预期：可以看到来源平台、原始链接、导入方式和当前解析状态。

### 预期结果

- 飞书和有道云都有 V1 可用导入路径。
- 平台受限时，系统不会直接失败，而是给出兜底方案。
- 多种导入证据可以合并到同一源资料。

### 边界场景

- 用户取消授权时，源链接仍保留。
- 导出文件和链接重复时，系统合并。
- 平台返回权限错误时，错误提示可理解。

## 6. 场景 4：本地解析失败到大模型兜底

### 测试目标

验证本地解析失败或质量低时，系统可以调用用户配置的大模型兜底，并把大模型结果保存为可复用解析改进产物。

### 起始条件

- 用户已配置至少一个大模型供应商。
- 用户选择“平衡”模式。
- 用户导入一个本地解析容易失败的文件或截图。

### 用户角色

AI 重度用户。

### 测试步骤

1. 用户导入文件。
   - 预期：系统创建源资料记录。
2. 本地解析器尝试解析。
   - 预期：解析状态进入“解析中”。
3. 本地解析失败或置信度低。
   - 预期：状态变为“大模型兜底等待中”。
4. 系统通过 ModelRouter 选择模型。
   - 预期：调用日志记录供应商、模型、来源、任务目的和内容大小。
5. 大模型返回结构化解析结果。
   - 预期：系统保存解析结果，并更新为“兜底成功”。
6. 系统生成解析改进产物。
   - 预期：保存失败模式、样例、规则或模板。
7. 用户重新导入同类内容。
   - 预期：系统优先尝试本地规则，再决定是否调用大模型。

### 预期结果

- 本地解析失败不会卡死流程。
- 大模型兜底结果进入记忆系统。
- 大模型结果可以反哺本地解析规则。
- 模型调用可追踪。

### 边界场景

- 用户未配置模型时，系统提示只能保留源资料和本地失败结果。
- 大模型调用失败时，显示失败原因并允许重试。
- 用户选择省 token 模式时，不自动调用外部模型。

## 7. 场景 5：有大模型问答到引用追溯

### 测试目标

验证配置大模型后，用户可以基于本地记忆问答，且答案必须带来源引用并可追溯。

### 起始条件

- 系统中已有多条已入记忆的资料。
- 用户已配置大模型。
- 污染内容默认被过滤。

### 用户角色

AI 重度用户。

### 测试步骤

1. 用户进入问答页。
   - 预期：页面显示问题输入框和模型状态。
2. 用户提问：“我最近关于图谱设计讨论了什么？”
   - 预期：系统先进行检索。
3. 系统召回相关 chunk。
   - 预期：过滤污染和隔离内容。
4. 系统调用大模型生成答案。
   - 预期：答案包含引用来源。
5. 用户点击一个引用。
   - 预期：打开源资料详情，并定位到对应页码、片段或时间点。
6. 用户查看使用过的资料。
   - 预期：页面展示使用到的 chunk 和来源。

### 预期结果

- 问答基于本地记忆。
- 答案带引用。
- 引用可以追溯到源资料。
- 无足够资料时，系统明确说明资料不足。

### 边界场景

- 大模型超时后，系统保留检索结果并提示重试。
- 检索结果为空时，不生成编造答案。
- 源文件已删除时，引用显示追溯状态异常。

## 8. 场景 6：错误引用到污染隔离

### 测试目标

验证用户可以把错误引用或错误记忆标记为污染，并确保污染内容不再参与普通搜索、问答和图谱推荐。

### 起始条件

- 系统中已有问答答案和引用。
- 至少一条引用来自错误 OCR 或过期内容。

### 用户角色

治理型用户。

### 测试步骤

1. 用户在问答答案中发现错误引用。
   - 预期：引用旁边有“标记错误/污染”入口。
2. 用户点击标记污染。
   - 预期：系统要求选择污染原因。
3. 用户选择“解析错误”。
   - 预期：该 chunk 或源资料进入隔离状态。
4. 用户再次进行同一问题问答。
   - 预期：系统不再使用被隔离内容。
5. 用户进入治理页面。
   - 预期：可以看到隔离内容和原因。
6. 用户选择恢复。
   - 预期：内容恢复为可用状态，必要时提示重建向量或图谱。

### 预期结果

- 错误引用可以被治理。
- 污染内容默认退出普通检索和问答。
- 用户可以查看和恢复隔离内容。

### 边界场景

- 用户标记整份源资料污染时，该源资料的所有派生记忆都应被排除。
- 用户只标记单个 chunk 污染时，不应影响其他正常 chunk。
- 恢复时如果派生产物已删除，应提示重建。

## 9. 场景 7：删除源文件并选择派生产物范围

### 测试目标

验证用户删除源文件时，可以控制是否删除解析文本、向量和图谱。

### 起始条件

- 系统中已有一份已解析、已入记忆、已进入图谱的源资料。

### 用户角色

治理型用户。

### 测试步骤

1. 用户在源资料详情点击删除。
   - 预期：系统弹出删除范围选择。
2. 用户选择“仅删除源文件，保留记忆”。
   - 预期：源文件删除，记忆仍可搜索，但追溯状态显示源文件缺失。
3. 用户撤销或对另一份资料选择“删除源文件、向量和图谱”。
   - 预期：源文件、向量索引和图谱节点/关系被删除或移除。
4. 用户查看搜索结果。
   - 预期：被删除向量不再出现。
5. 用户查看图谱。
   - 预期：被删除图谱节点和关系不再显示。

### 预期结果

- 删除源文件时不会悄悄留下不可控派生产物。
- 用户可以选择保留或删除记忆。
- 删除行为有记录。

### 边界场景

- 默认操作应为移入回收站并从检索排除。
- 执行硬删除前需要二次确认。
- 删除失败时不应出现半删除状态，或必须显示待修复状态。

## 10. 场景 8：图谱搜索到源文件追溯

### 测试目标

验证图谱页面可搜索、可定位、可点回源资料，并能过滤污染节点。

### 起始条件

- 系统中已有至少 50 个图谱节点。
- 节点包含源文件、主题、项目、人物、想法。
- 存在一条污染节点。

### 用户角色

AI 重度用户。

### 测试步骤

1. 用户打开图谱页。
   - 预期：图谱加载并显示节点和关系。
2. 用户搜索“本地 AI 记忆”。
   - 预期：命中节点高亮，图谱自动聚焦。
3. 用户点击命中节点。
   - 预期：右侧显示节点详情、关系来源和相邻节点。
4. 用户点击“查看源资料”。
   - 预期：打开对应源资料详情。
5. 用户切换“显示污染节点”。
   - 预期：污染节点显示特殊标记。
6. 用户关闭污染节点显示。
   - 预期：污染节点从图谱中隐藏。

### 预期结果

- 图谱不是装饰，而是可搜索、可定位、可追溯。
- 关系来源可解释。
- 污染节点不会默认干扰探索。

### 边界场景

- 节点过多时，搜索仍能在可接受时间内定位。
- 节点没有源资料时，应显示原因。
- 图谱加载失败时，提供列表兜底。

验证用户可以导出图谱快照，并避免泄露敏感路径或私密来源。

### 起始条件

- 图谱页已加载。

### 用户角色

AI 重度用户。

1. 用户点击“导出快照”。
   - 预期：系统显示导出选项。

   - 预期：导出预览不显示本地路径。

   - 预期：系统生成图片文件。

## 12. 场景 10：Codex/MCP 调用本地记忆

### 测试目标

验证外部 AI 工具可以通过 MCP 安全调用本地记忆，并且调用有权限、日志和污染过滤。

### 起始条件

- 用户已启用 MCP Server。
- MCP 只监听本地地址。
- 系统中已有已入记忆资料。
- 存在污染或隔离内容。

### 用户角色

开发者用户。

### 测试步骤

1. 用户在设置页启用 MCP Server。
   - 预期：系统显示连接信息和可用工具列表。
2. 用户在 Codex 或外部 AI 工具中配置 MCP。
   - 预期：外部工具可以连接本地 MCP。
3. 外部工具调用 `memory.search`。
   - 预期：返回记忆标题、片段、来源和状态。
4. 外部工具调用 `memory.get_context`。
   - 预期：返回结构化上下文、chunk 和引用来源。
5. 系统检查返回内容。
   - 预期：污染或隔离内容不返回。
6. 用户查看调用日志。
   - 预期：可以看到调用时间、工具名、请求类型和返回范围。
7. 用户关闭 MCP Server。
   - 预期：外部工具无法继续调用。

### 预期结果

- 外部 AI 工具可以安全调用本地记忆。
- 调用默认本地可控。
- 污染数据不泄露。
- 调用可审计。

### 边界场景

- 未启用 MCP 时，外部工具连接失败并给出明确提示。
- 外部工具请求过大时，系统限制返回内容。
- 用户删除某条记忆后，外部工具不能再获取该内容。

## 13. 场景 11：模型供应商配置与切换

### 测试目标

验证用户可以配置主流模型供应商、自定义兼容 endpoint 和本地模型，并按任务选择模型。

### 起始条件

- 用户打开设置页。
- 用户拥有至少一个模型 API Key，或本地已运行 Ollama/LM Studio。

### 用户角色

AI 重度用户。

### 测试步骤

1. 用户进入模型设置页。
   - 预期：页面显示供应商列表和添加入口。
2. 用户添加 OpenAI-Compatible Endpoint。
   - 预期：可填写 Base URL、API Key 和模型名。
3. 用户点击测试连接。
   - 预期：系统显示连接成功或失败原因。
4. 用户设置默认问答模型和解析兜底模型。
   - 预期：TaskModelPolicy 保存成功。
5. 用户切换到省 token 模式。
   - 预期：解析失败时不自动调用外部模型。
6. 用户切换到平衡模式。
   - 预期：解析失败时可触发大模型兜底。

### 预期结果

- 模型配置不绑定单一供应商。
- 自定义 endpoint 可用。
- 不同任务可以使用不同模型。
- 调用策略影响实际行为。

### 边界场景

- API Key 错误时，系统不得保存为可用状态。
- 供应商不可达时，显示网络或认证错误。
- 用户删除默认模型时，相关任务策略应提示重新选择。

## 14. 场景 12：完整 V1 演示路径

### 测试目标

验证 V1 的完整产品故事可以在一条演示路径中跑通。

### 起始条件

- 应用新安装。
- 用户准备 PDF、截图、网页链接、飞书文档、有道云内容。
- 用户可选择配置或不配置大模型。

### 用户角色

综合用户。

### 测试步骤

1. 用户第一次启动应用并选择数据目录。
2. 用户导入 PDF、截图、网页链接、飞书文档和有道云内容。
3. 用户查看源资料库，确认每条资料的状态。
4. 系统本地解析资料。
5. 本地解析失败的资料进入大模型兜底。
6. 资料进入向量记忆。
7. 用户在无模型模式下搜索资料。
8. 用户配置模型后进行问答。
9. 用户点击答案引用回到源资料。
10. 用户标记一条错误引用为污染。
11. 用户确认污染内容不再参与搜索和问答。
12. 用户打开图谱，搜索节点并点回源文件。
13. 用户导出图谱快照。
14. 用户启用 MCP，让 Codex 搜索本地记忆。

### 预期结果

- 6 个核心卖点全部跑通。
- 任何一步失败都能看到明确原因。
- 用户始终知道资料在哪里、是否已解析、是否已入记忆、是否可被 AI 使用。
