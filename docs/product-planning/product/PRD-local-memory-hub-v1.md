# PRD：Local Memory Hub V1

## 1. 概要

Local Memory Hub V1 是一个本地优先的个人源资料库与 AI 记忆系统。它负责保存源文件、导入外部文档、本地解析内容，并在本地解析失败时调用大模型兜底，再把清洗后的内容转成可搜索的记忆、向量索引和可视化图谱。

第一版必须让用户明确感受到三件事：源数据是安全可控的，系统清楚知道哪些内容已经进入记忆，不想要或污染的记忆可以被移除。

## 2. 相关角色

| 名称 | 角色 | 说明 |
| --- | --- | --- |
| 创始人 / 产品负责人 | 产品方向 | 负责产品范围和验证 |
| 工程负责人 | 桌面端、本地服务、解析管线 | 负责 Mac/Windows 一键启动 |
| AI / 数据管线负责人 | 解析、清洗、向量、图谱 | 负责本地优先和大模型兜底闭环 |
| 设计负责人 | 源资料库、图谱、 onboarding | 负责可用性和视觉识别 |

## 3. 背景

用户每天会从很多地方接收有价值的信息：文件、截图、视频、网页、飞书文档、有道云笔记、聊天内容和各种链接。这些信息非常分散。传统文件夹可以保存文件，但不能理解内容。AI 工具可以回答问题，但通常看不到用户真实的源资料库。

这个产品由两层组成：

- 源资料层：保存原始文件和外部文档，提供文件夹、日期、类型、来源、解析状态、入记忆状态和追溯能力。
- AI 记忆层：保存解析后的文本、文本片段、向量索引、图谱节点、图谱关系、摘要和引用。

用户必须始终能够从一条 AI 记忆追溯回源文件，也必须能够把污染内容从搜索、图谱和问答中移除。

## 4. 目标

### 4.1 产品目标

构建一个本地优先的应用，让用户低成本导入源数据，通过本地优先的解析闭环处理内容，并把它变成可信的 AI 记忆。

### 4.2 为什么重要

这个产品不应该像另一个普通知识库。它应该像一个可靠的本地记忆工作台：用户把信息丢进来，可以看到它是否已保存、是否解析成功、是否进入记忆，之后可以通过搜索、图谱或 Codex 这类 AI 工具重新使用这些资料。

### 4.3 关键结果

| 关键结果 | 目标 |
| --- | --- |
| 源资料导入成功率 | 90% 的受支持来源类型可以保存进源资料库 |
| 状态可见性 | 100% 的源资料都显示解析状态和入记忆状态 |
| 记忆可追溯 | 使用记忆回答时，100% 的 AI 答案可以引用源资料 |
| 污染控制 | 被标记污染的记忆可在 5 秒内从检索和图谱中排除 |
| 大模型兜底闭环 | 至少 3 类解析失败可以通过大模型结果修复或改善 |
| 外部文档导入 | V1 必须提供飞书和有道云的导入路径，即使部分场景需要手动授权或导出兜底 |
| 升级数据保留 | 应用升级后，用户原有源资料、记忆、图谱和配置必须保留 |

## 5. 目标用户

### 5.1 早期核心用户

- 使用 Codex、Cursor、Claude、ChatGPT 等 AI 工具的开发者。
- 收集调研、文档、截图和会议资料的产品经理与创业者。
- 收集视频、文章、笔记和参考资料的内容创作者。
- 处理 PDF、网页和外部笔记的研究者与学生。

### 5.2 用户约束

- 不想手动整理每一条资料。
- 关心本地控制和隐私。
- 需要知道一个文件是否真的能被 AI 使用。
- 可能使用飞书、有道云笔记等外部文档平台。
- 希望图谱不仅有用，还要有视觉冲击力，并具备一定社交传播属性。

## 6. 价值主张

### 6.1 核心价值

把源数据保存在本地，清楚显示它是否已经进入记忆，并让它可以被搜索、追溯、删除和 AI 调用。

### 6.2 用户收益

- 在一个地方找到原始文件和导入的外部文档。
- 清楚看到每条资料的保存、解析、索引、失败、污染或删除状态。
- AI 回答带有源资料引用。
- 通过可视化图谱探索主题、文件、想法和关系。
- 本地解析能力会随着大模型兜底结果逐步改善。
- 可以让 Codex 或其他模型通过 API/MCP 使用本地记忆。

### 6.3 解决的痛点

- 不再猜测一个文件是否已经解析。
- 不再接受没有来源的 AI 回答。
- 错误 OCR、错误转写、过期资料或错误摘要不会永久污染记忆。
- 不需要把每一份资料手动复制到新的笔记系统。

## 7. 方案

### 7.1 页面和用户体验

#### 首次启动引导

安装完成后，系统必须提供首次启动引导。

引导步骤：

1. 选择本地数据目录。
2. 配置大模型，可跳过。
3. 随便导入一段文本。
4. 系统保存、解析并生成第一条记忆。
5. 进入图谱首页，默认聚焦第一条记忆节点。

配置大模型步骤应支持主流模型供应商和自定义兼容接口。用户可以跳过该步骤；跳过后，搜索和源资料管理仍然可用。

导入文本步骤应提供：

- 粘贴文本
- 使用示例文本
- 跳过

导入成功后展示：

- 已保存
- 解析成功
- 已入记忆
- 已生成图谱节点

#### 图谱首页

应用打开后的默认首页是图谱首页。

图谱首页需要像 Obsidian 图谱一样，第一眼展示用户资料和记忆之间的关系，同时支持搜索、快捷导入、节点详情和源资料追溯。

图谱首页默认展示：

- 最近导入的源资料节点
- 最近生成的主题节点
- 最近问答引用过的记忆节点
- 高关联度的想法/项目节点
- 当前污染节点默认隐藏

主要操作：

- 搜索图谱节点
- 粘贴链接
- 上传文件
- 导入一段文本
- 询问本地记忆
- 打开源资料
- 标记污染

#### 工作台

工作台展示正在解析的任务、最近源资料、最近记忆召回、污染提醒和外部调用状态。

主要操作：

- 查看处理任务
- 查看失败项
- 查看污染提醒
- 询问本地记忆
- 打开图谱

#### Inbox

所有新导入的资料先进入 Inbox。

每条资料显示：

- 标题
- 来源类型
- 来源平台
- 导入时间
- 解析状态
- 入记忆状态
- 追溯状态
- 污染状态
- 主要操作

#### 源资料库

这是用户可信任的本地源文件管理器。

必备字段：

- 文件夹
- 文件名
- 日期
- 类型
- 来源
- 源路径或源链接
- 解析状态
- 解析成功或失败原因
- 是否已入记忆
- 可追溯状态
- 污染状态
- 源文件检索

必备操作：

- 打开源文件
- 预览源文件
- 重新解析
- 加入记忆
- 从记忆中移除
- 标记为污染
- 移入回收站
- 高级删除源文件

#### 记忆库

展示 AI 可以使用的记忆产物。

必备字段：

- 记忆标题
- 对应源资料
- 文本片段数量
- 向量索引状态
- 图谱状态
- 摘要状态
- 最近被 AI 使用时间
- 污染状态

必备操作：

- 打开源文件
- 查看文本片段
- 从向量检索中移除
- 从图谱中移除
- 标记污染
- 重建向量索引
- 重建图谱关系

#### 图谱

图谱页面应具有 Obsidian 风格，但要更有社交感、更鲜活、更有视觉冲击力。

必备节点类型：

- 源文件
- 外部文档
- 主题
- 实体
- 项目
- 人物
- 想法
- 问题
- 记忆

必备关系类型：

- 来自同一源资料
- 属于同一主题
- 语义相似
- 时间接近
- 用户手动创建关系
- 大模型推断关系
- 引用关系

每条关系都必须显示它为什么存在。

图谱要求：

- 平滑缩放和平移
- 节点拖拽
- hover 高亮
- 选中节点后高亮一跳关系
- 节点聚类
- 时间线模式
- 来源筛选
- 入记忆状态筛选
- 污染状态筛选
- 关系类型筛选
- 点击节点查看源文件和记忆
- 删除污染节点或污染关系
- 后续支持分享/导出图谱快照

节点跟随鼠标移动、靠近避让、局部力场扰动等高级互动不进入 V1 必须范围，后续在图谱库 Spike 后再决定是否实现。

V1 的图谱应有独特视觉风格。它可以通过可分享的洞察具备社交属性，但 V1 不需要做完整社交网络。

#### 问答

AI 问答页面必须展示：

- 答案
- 使用到的源资料
- 使用到的文本片段
- 通过图谱扩展到的节点
- 置信提示
- 标记引用错误或污染的按钮

#### 设置

必备设置：

- 数据目录
- 数据目录位置展示
- 应用升级和数据迁移状态
- API Key
- 主流模型供应商
- 自定义 OpenAI-Compatible / Anthropic-Compatible Endpoint
- 本地模型地址
- 外部文档连接器
- MCP Server
- 解析模式
- 隐私控制
- 备份与恢复
- 删除策略

### 7.2 核心功能

#### 功能 1：一键启动本地应用

应用需要用一套核心代码运行在 Mac 和 Windows，并提供两种安装入口：

推荐桌面形态：

- Tauri 桌面壳
- React / Next.js / Vite UI
- 本地 API 服务
- 本地 worker
- 本地数据库
- 本地文件存储

推荐 npm 形态：

```bash
npm install -g local-memory-hub
lmh start
```

或：

```bash
npx local-memory-hub start
```

npm 模式启动后需要自动：

- 检查 Node 版本。
- 创建或读取本地数据目录。
- 启动本地 API 服务。
- 启动 Web UI。
- 可选启动 MCP-like 服务。
- 打开浏览器到本地页面。
- 在终端展示数据目录、端口和关闭方式。

npm 模式面向开发者和早期用户，不替代普通用户的 Mac/Windows 桌面安装包。

npm 模式 V1 命令：

| 命令 | 作用 |
| --- | --- |
| `lmh start` | 启动本地 API、Web UI 和可选 MCP-like 服务 |
| `lmh stop` | 停止由 CLI 启动的本地服务 |
| `lmh open` | 打开本地 Web UI |
| `lmh doctor` | 检查 Node 版本、端口、数据目录权限、服务状态和模型配置 |
| `lmh data-dir` | 显示当前数据目录来源和路径 |
| `lmh config` | 查看或修改基础配置 |
| `lmh migrate` | 手动触发 schema 检查和迁移 |
| `lmh version` | 显示 CLI、API、数据 schema 版本 |

`lmh start` 参数至少包括：

- `--data-dir`
- `--port`
- `--host`
- `--no-open`
- `--with-mcp`

npm 模式首次启动如果没有数据目录配置，应使用默认目录并在终端明确展示；如果用户传入 `--data-dir`，优先使用该目录。

#### 功能 2：源资料库

每一次导入在开始解析之前，都必须先创建源资料记录。

源资料状态：

- 已保存
- 等待解析
- 解析中
- 解析成功
- 部分解析成功
- 解析失败
- 等待入记忆
- 已入记忆
- 部分入记忆
- 入记忆失败
- 已污染
- 已隔离
- 已删除

#### 功能 3：外部文档导入

V1 必须包含外部文档导入路径。

优先级：

1. 飞书
2. 腾讯文档
3. 有道云笔记
4. 通用网页文档 URL
5. 手动导出/上传兜底

飞书 V1 方案：

- 粘贴飞书文档链接
- 在可行时支持 OAuth 或 token 连接器
- 在平台权限允许时支持事件订阅或定时同步
- 支持手动导出上传兜底
- 保留源链接和平台元数据

腾讯文档 V1 方案：

- 粘贴腾讯文档链接
- 在可行时支持 OAuth 连接器
- 通过定时轮询检查变更，不把实时 Webhook 同步作为 V1 必须承诺
- 支持手动导出上传兜底
- 保留源链接、文档 ID、标题、授权账号和最近同步时间

有道云 V1 方案：

- 在可访问时支持粘贴笔记链接
- 支持导出文件导入
- 支持复制内容导入
- 在可行时保留源链接、笔记本、标题等元数据
- 不把自动同步作为 V1 默认承诺；有道云官方 OpenAPI 停止新增申请时，优先走导出/复制兜底

如果平台 API 权限受限或不稳定，产品仍应保存源链接，并提示用户通过导出或上传完成导入。

#### 功能 4：本地优先解析闭环

解析不能只是一次性流水线，必须是可学习的闭环。

流程：

1. 优先尝试本地解析器。
2. 如果本地解析成功，进行清洗和入库。
3. 如果本地解析失败或质量过低，调用用户配置的大模型。
4. 把大模型解析结果保存为结构化修正结果。
5. 对比本地解析输出和大模型输出。
6. 生成可复用的解析规则、模板或样例。
7. 下次遇到同类内容时，先使用新规则，再决定是否调用大模型。
8. 追踪新规则是否改善了后续解析效果。

这个闭环的目标是减少未来 token 消耗，但在需要时不拒绝使用大模型。

解析改进产物：

- 失败模式
- 来源类型
- 原始样本 hash
- 本地解析输出
- 大模型修正输出
- 生成的规则
- 规则置信度
- 最近应用时间
- 成功次数
- 失败次数

#### 功能 5：成本感知的大模型兜底

应用支持三种模式：

- 省 token：除非用户主动要求，否则只做本地处理。
- 平衡：本地解析失败或置信度低时调用大模型。
- 深度：更多使用大模型生成结构、摘要和图谱关系。

默认模式为平衡。

在内容发送给外部模型之前，应用应展示或记录：

- 供应商
- 模型
- 来源
- 近似内容大小
- 调用目的
- 调用结果

#### 功能 6：记忆污染控制

污染来源：

- OCR 错误
- 转写错误
- 文档解析错误
- 内容过期
- 重复内容
- AI 摘要错误
- 用户不想让该源资料被 AI 使用

操作：

- 标记污染
- 隔离
- 恢复
- 从向量检索中移除
- 从图谱中移除
- 移除摘要
- 删除派生产物
- 全部删除

删除源文件时必须提供：

- 仅删除源文件，保留记忆
- 删除源文件和解析文本
- 删除源文件、向量和图谱
- 全部删除
- 移入回收站

默认操作：移入回收站，并从检索中排除。

#### 功能 7：向量 + 图谱联合检索

检索流程：

1. 对用户问题生成 向量索引。
2. 从向量索引中召回相关 文本片段。
3. 将 文本片段 映射到图谱节点。
4. 沿图谱扩展一跳或两跳。
5. 过滤污染或隔离数据。
6. 对结果重新排序。
7. 生成带引用的回答。

图谱关系必须显示来源。关系来源可以是：

- 规则生成
- 向量相似生成
- 大模型推断
- 用户手动创建

#### 功能 8：API 和 MCP

V1 应暴露本地 API 和 MCP 工具。

必备工具：

- `memory.search`
- `memory.add`
- `memory.get_context`
- `source.import`
- `source.status`
- `source.mark_polluted`
- `graph.search`
- `graph.get_neighbors`

### 7.3 技术说明

建议技术栈：

- 桌面端：Tauri
- UI：React / Next.js / Vite
- 本地服务：FastAPI 或 Node
- 数据库：PostgreSQL + pgvector，轻量本地版可考虑 SQLite
- 图谱存储 V1：关系型节点/边表
- 文件存储：本地数据目录
- worker：本地任务队列
- OCR：PaddleOCR 或同类方案
- 视频/音频：FFmpeg + 本地 Whisper 引擎
- 外部文档连接器：插件式 connector 层
- MCP：本地 MCP Server

数据模型必须分离：

- 源资料记录
- 原始文件
- 抽取文本
- 文本片段
- 向量索引
- 图谱节点
- 图谱关系
- 摘要
- 解析规则
- 删除记录

### 7.4 假设

高风险假设：

- 用户能理解源资料库和记忆库的区别。
- 用户对飞书、有道云导入的需求强到值得放进 V1。
- 大模型兜底能改善后续本地解析，而不是变成隐藏成本黑洞。
- 图谱既能有用，也能足够好看。
- 污染删除机制不会让用户觉得复杂。
- 在包含解析器、连接器、向量、图谱和 MCP 的情况下，本地应用仍能保持简单启动。

- 完整社交网络
- 自动读取完整微信聊天记录
- 全天候录屏
- 团队权限系统
- 完整视频视觉理解
- 完美覆盖所有外部平台
- 云同步
