# Pre-Mortem：Local Memory Hub V1

## 1. 前提

假设 Local Memory Hub V1 在 14 天后发布，但发布后失败了。

失败表现：

- 用户安装后没有持续使用。
- 用户导入资料后不知道是否成功。
- 图谱看起来很酷，但用户很快不用。
- 搜索和问答结果不可信。
- 本地解析失败率高。
- 大模型配置复杂，用户卡在 onboarding。
- MCP/API 价值没有被开发者感知。

本文件倒推最可能的失败原因，并提出预防动作。

## 2. Tigers：真实风险

## Tiger 1：图谱首页炫酷但不可用

**紧急程度：Launch-Blocking**

### 风险

产品默认首页是图谱。如果图谱只是漂亮动画，但不能搜索、定位、点回源文件、解释关系，用户会觉得它是装饰。

### 为什么真实

很多知识图谱产品都死在“看起来很聪明，但无法完成任务”。默认首页给图谱，意味着图谱承担第一印象和主工作台职责。

### 缓解动作

- 图谱 V1 必须支持搜索节点。
- 图谱节点必须能打开源资料。
- 图谱关系必须显示原因。
- 空状态不能伪造数据。
- 图谱加载失败时必须有列表兜底。

### Owner

设计 + 前端 + 图谱工程。

## Tiger 2：首次引导卡在大模型配置

**紧急程度：Launch-Blocking**

### 风险

用户安装后第二步配置大模型，如果 API Key、供应商、Base URL、模型名让用户困惑，用户可能直接流失。

### 为什么真实

大模型配置对开发者还好，对普通知识工作者比较重。哪怕目标用户偏技术，也不能让首启体验像填云服务控制台。

### 缓解动作

- 大模型配置必须可跳过。
- 跳过后搜索和源资料管理仍可用。
- 引导第三步必须让用户导入一段文本并看到图谱节点。
- 模型配置页提供“测试连接”。
- 自定义 endpoint 放到高级入口，不挡住主流程。

### Owner

产品 + 设计 + 模型集成。

## Tiger 3：导入系统变成平台 if/else 屎山

**紧急程度：Launch-Blocking**

### 风险

飞书、有道、链接、截图、快捷键、MCP、视频平台都要导入。如果没有统一导入模型和插件机制，代码会快速失控。

### 为什么真实

导入入口会持续增加。每个入口都有认证、失败、重试、去重、状态和兜底逻辑。

### 缓解动作

- 必须先实现 `ImportRequest`。
- 必须先实现 `ImporterRegistry`。
- 快捷导入不得绕过导入管线。
- 外部平台必须通过 Adapter。
- 所有导入都先生成 `SourceRecord`。

### Owner

后端 / 本地服务工程。

## Tiger 4：源资料状态和记忆状态混淆

**紧急程度：Launch-Blocking**

### 风险

用户看到“成功”但不知道是导入成功、解析成功，还是入记忆成功。之后问不到内容，就会觉得系统坏了。

### 为什么真实

这个产品的核心信任来自状态透明。如果状态混乱，用户会很快失去控制感。

### 缓解动作

- UI 分开显示导入状态、解析状态、入记忆状态、污染状态、追溯状态。
- “解析成功”不能等同于“已入记忆”。
- 所有失败状态可点击查看原因。
- 状态标签统一由 `StatusBadge` 管理。

### Owner

产品 + 设计 + 前端 + 数据模型。

## Tiger 5：本地解析不稳定，导致资料无法变成记忆

**紧急程度：Launch-Blocking**

### 风险

用户导入 PDF、截图、外部文档后，解析失败或质量差，系统无法进入搜索和问答闭环。

### 为什么真实

本地解析涉及 PDF、OCR、网页抽取、视频转写、外部文档导出格式。失败率天然不低。

### 缓解动作

- V1 先明确支持类型和限制。
- 所有解析失败必须保存源资料和失败原因。
- 解析失败可触发大模型兜底。
- 大模型兜底结果先保存为样例，不一开始强求自动规则完美生成。
- 建立 100 个真实样本解析基准。

### Owner

AI / 数据管线工程。

## Tiger 6：大模型兜底变成 token 黑洞

**紧急程度：Launch-Blocking**

### 风险

系统频繁把解析、摘要、图谱都交给外部模型，用户成本不可控，也会产生隐私焦虑。

### 为什么真实

“本地失败就调用大模型”如果没有策略，很容易成为隐藏成本黑洞。

### 缓解动作

- 默认模式为“平衡”。
- 支持“省 token / 平衡 / 深度”。
- 外部调用必须记录供应商、模型、来源、内容大小和目的。
- 大模型结果必须反哺本地规则或样例。
- UI 显示模型调用日志。

### Owner

模型系统 + 产品 + 隐私设计。

## Tiger 7：无模型兜底体验太弱

**紧急程度：Fast-Follow**

### 风险

用户跳过模型配置后，只能看到一堆搜索结果，感觉产品没有 AI 价值。

### 为什么真实

我们强调“无模型也可用”，但用户可能期望它像 AI 产品一样给出总结。

### 缓解动作

- 无模型模式要明确说明是搜索兜底。
- 搜索结果要有命中片段、来源、日期、状态和图谱节点。
- 提供“配置模型后生成答案”入口。
- 可以用本地小模型做轻量摘要，但必须标明。

### Owner

搜索体验 + 模型设置。

## Tiger 8：污染治理太复杂，用户不敢删

**紧急程度：Launch-Blocking**

### 风险

删除源文件时有解析文本、向量索引、图谱节点、历史引用等派生产物。用户看不懂删除范围，不敢操作，或者误删后崩溃。

### 为什么真实

数据生命周期是复杂产品的坑。越本地优先，用户越在意控制权。

### 缓解动作

- 默认操作是移入回收站并从检索排除。
- 删除弹窗必须解释影响范围。
- “仅删除源文件，保留记忆”等选项要有清楚说明。
- 污染先隔离，不默认硬删除。
- 所有删除有记录和可恢复路径。

### Owner

产品 + 设计 + 数据模型。

## Tiger 9：MCP/API 接入做了，但开发者感知不到价值

**紧急程度：Fast-Follow**

### 风险

外部 AI 工具能调用本地记忆，但用户不知道怎么接 Codex，也不知道什么时候该用 `memory.search`、`memory.get_context`。

### 为什么真实

开发者能力如果没有文档、示例和 demo，很容易变成隐藏功能。

### 缓解动作

- 外部接入页提供 Codex 配置复制。
- 提供最小 demo：Codex 查询本地记忆并引用源资料。
- 文档解释 `memory.ask` 和 `memory.get_context` 的区别。
- 所有外部调用有日志。

### Owner

开发者体验 + 文档。

## Tiger 10：V1 范围太大，团队无法完成

**紧急程度：Launch-Blocking**

### 风险

V1 同时包含导入、解析、模型、搜索、问答、污染、图谱、MCP、外部平台。范围很大，任何一条线失控都会拖垮整体。

### 为什么真实

6 个核心卖点都必须有，但每个卖点内部都能展开成大产品。

### 缓解动作

- 每个核心卖点只做最小可闭环版本。
- 先保证第一条演示路径跑通。
- 图谱先做搜索、定位、追溯，不做复杂编辑器。
- 飞书/有道先做导入路径和兜底，不追求完美同步。
- 大模型反哺先保存样例，不追求自动生成复杂规则引擎。

### Owner

产品负责人 + 技术负责人。

## 3. Paper Tigers：看似吓人但被高估的风险

## Paper Tiger 1：必须一开始支持所有模型供应商原生 API

### 判断

不需要。

### 原因

V1 可以优先支持主流 Provider 和 OpenAI-Compatible / Anthropic-Compatible。很多国内模型可以先通过兼容接口接入，后续再做原生 Adapter。

## Paper Tiger 2：必须一开始做完整社交网络

### 判断

不需要。

### 原因

图谱需要社交传播属性，但 V1 只需要导出快照和视觉记忆点。关注、评论、分享广场都可以后置。

## Paper Tiger 3：必须一开始支持全自动微信读取

### 判断

不需要。

### 原因

微信全自动读取有技术、隐私和合规压力。V1 可以通过复制、截图、文件拖拽、手动导入解决。

## Paper Tiger 4：必须做云同步

### 判断

不需要。

### 原因

产品核心定位是本地优先。云同步会引入账号、加密、冲突合并、隐私承诺，V1 可以后置。

## 4. Elephants：还没讨论够的隐忧

## Elephant 1：普通用户是否真的理解“源资料”和“记忆”的区别

### 为什么需要验证

这是整个产品的信息架构基础。如果用户理解不了，会觉得页面太多、概念太多。

### 调查方式

用低保真原型做 5-8 人可用性测试，观察用户能否完成：

- 找源文件。
- 判断是否已入记忆。
- 删除源文件但保留记忆。
- 从问答引用回到源资料。

## Elephant 2：图谱作为默认首页是否会压过导入和搜索

### 为什么需要验证

图谱首页有传播感，但用户刚进入时最常用动作可能是导入和搜索。如果图谱太抢戏，效率会下降。

### 调查方式

测试两版首页：

- A：图谱首页。
- B：工作台首页。

比较：

- 首次导入完成率。
- 搜索入口发现率。
- 用户主观兴奋度。
- 用户是否理解产品价值。

## Elephant 3：大模型反哺本地解析规则的效果是否足够好

### 为什么需要验证

这是产品很有想象力的地方，但工程上不一定一开始就能稳定工作。

### 调查方式

先选 3 类失败：

- OCR 结构错乱。
- PDF 标题层级错乱。
- 外部文档导出格式混乱。

用大模型生成修正样例，观察下次同类样本是否减少调用。

## Elephant 4：外部文档导入的合规和权限边界

### 为什么需要验证

飞书、有道云可能有 API 权限、导出限制、企业空间权限等问题。

### 调查方式

优先调研实际 API 和用户常见导出路径。V1 只承诺“导入路径”，不承诺全自动同步。

## Elephant 5：本地一键启动的技术复杂度

### 为什么需要验证

Tauri + 本地服务 + 本地数据库 + worker + OCR/Whisper + MCP 组合复杂。

### 调查方式

先做技术 Spike：

- Tauri 启动本地服务。
- 本地数据目录。
- 文件导入。
- 简单搜索。
- MCP demo。

## 5. Launch-Blocking Tigers 行动计划

| 风险 | 缓解动作 | Owner | 完成节点 |
| --- | --- | --- | --- |
| 图谱首页炫酷但不可用 | 图谱必须支持搜索、定位、点回源文件、关系原因 | 设计 + 前端 | UI 原型完成前 |
| 首次引导卡在模型配置 | 模型配置可跳过，引导第三步导入文本 | 产品 + 设计 | 低保真原型完成前 |
| 导入系统变屎山 | 先实现 ImportRequest、ImporterRegistry、Import Pipeline | 后端/本地服务 | 代码开发前 |
| 状态混淆 | 分开显示导入、解析、入记忆、污染、追溯状态 | 产品 + 前端 | 组件设计完成前 |
| 本地解析不稳定 | 建立解析样本集和失败兜底流程 | 数据管线 | 技术 Spike 阶段 |
| 大模型兜底 token 黑洞 | 三档模式、调用日志、结果反哺规则 | 模型系统 | 技术 Spike 阶段 |
| 污染治理太复杂 | 默认隔离、影响范围说明、可恢复 | 产品 + 设计 | UI 原型完成前 |
| V1 范围太大 | 每个卖点只做最小闭环，冻结开发批次 | 产品负责人 | 开发排期前 |

## 6. 发布前必须验证

发布前必须跑通：

1. 首次引导：配置模型可跳过，导入一段文本，进入图谱首页。
2. 文件导入：保存源资料，解析成功，入记忆，搜索可用。
3. 解析失败：触发大模型兜底，记录调用日志。
4. 问答：答案带引用，并能点回源资料。
5. 污染：标记污染后不再参与普通搜索和问答。
6. 图谱：搜索节点，定位节点，点回源资料。
7. 删除：删除源文件时可选择是否删除向量索引和图谱。
8. MCP：Codex 能调用本地记忆，且调用有日志。

## 7. 结论

V1 最危险的地方不是技术单点不可行，而是范围和复杂度叠加。

必须坚持：

```text
每个核心卖点都要有，但每个核心卖点只做最小可信闭环。
```

图谱是第一印象，不能只是好看。状态是信任基础，不能模糊。导入是入口，不能写乱。污染治理是长期使用的保险，不能后补。
