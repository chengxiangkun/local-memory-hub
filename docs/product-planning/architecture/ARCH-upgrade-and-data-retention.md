# 升级与数据保留架构设计

## 1. 目标

Local Memory Hub 是本地优先产品。应用升级后，用户之前导入的源资料、解析结果、记忆、向量索引、图谱、模型配置和外部接入配置都不能丢失。

用户必须清楚知道：

- 数据保存在哪里。
- 升级是否会迁移数据。
- 迁移是否成功。
- 失败后如何恢复。
- 删除应用是否会删除数据。

## 2. 核心原则

### 2.1 应用和数据分离

应用目录和数据目录必须分离。

```text
应用目录：可以升级、替换、重装
数据目录：保持稳定，不随应用删除自动消失
```

### 2.2 用户可见数据目录

设置页必须展示当前数据目录。

操作：

- 查看数据目录
- 打开数据目录
- 更换数据目录
- 备份数据目录
- 查看数据占用

### 2.3 升级前自动检查

应用升级或首次打开新版本时，需要检查：

- 当前应用版本
- 当前数据 schema 版本
- 是否需要迁移
- 数据目录是否可写
- 是否需要备份

### 2.4 迁移必须可追踪

每次迁移都要记录：

- 旧版本
- 新版本
- 开始时间
- 结束时间
- 迁移步骤
- 成功或失败
- 错误日志

## 4. 版本管理

### 4.1 应用版本

记录当前应用版本：

```json
{
  "app_version": "0.1.0",
  "build": "2026.06.17",
  "updated_at": "2026-06-17T10:00:00Z"
}
```

### 4.2 Schema 版本

记录数据 schema 版本：

```json
{
  "schema_version": 3,
  "compatible_app_version": "0.1.0",
  "updated_at": "2026-06-17T10:00:00Z"
}
```

## 5. 迁移系统

### 5.1 Migration 接口

```ts
interface Migration {
  id: string;
  fromVersion: number;
  toVersion: number;
  description: string;

  up(context: MigrationContext): Promise<void>;
  down?(context: MigrationContext): Promise<void>;
  verify(context: MigrationContext): Promise<MigrationVerifyResult>;
}
```

### 5.2 迁移流程

```text
启动新版本应用
  ↓
读取数据目录 schema 版本
  ↓
检查是否需要迁移
  ↓
迁移前备份关键数据
  ↓
执行迁移
  ↓
校验迁移结果
  ↓
记录迁移历史
  ↓
进入应用
```

### 5.3 迁移前备份

迁移前至少备份：

- 数据库
- 配置文件
- 模型供应商配置
- 图谱元数据
- 向量索引元数据

原始源文件通常不复制完整备份，避免大文件消耗过高。但迁移不能修改原始源文件。

## 6. 数据兼容策略

### 6.1 原始源文件永不迁移破坏

`raw/` 下的源文件是最高优先级资产。

迁移可以新增索引、重建派生产物，但不能破坏原始文件。

### 6.2 派生产物可重建

这些数据可以重建：

- 抽取文本
- OCR 结果
- 转写结果
- 文本片段
- 向量索引
- 图谱节点和关系
- 摘要

因此迁移失败时，应优先保护源文件和源资料记录。

### 6.3 配置需要加密迁移

模型 API Key 和外部平台 token 必须加密保存。

迁移时不得把密钥写入明文日志。

## 7. 升级 UI

### 7.1 设置页展示

设置页需要展示：

```text
应用版本：0.1.0
数据版本：3
数据目录：/Users/xxx/LocalMemoryHub
数据占用：12.4 GB
最近迁移：成功
```

操作：

- 打开数据目录
- 备份数据
- 更换数据目录
- 查看迁移日志

### 7.2 升级提示

如果新版本需要迁移：

```text
新版本需要升级本地数据结构。

你的源文件不会被删除。
升级前会自动备份关键数据。

[开始升级] [稍后]
```

### 7.3 迁移失败提示

```text
数据升级失败。

源文件仍然保留。
系统已保留升级前备份。

[查看错误] [恢复备份] [重试]
```

## 8. 更换数据目录

用户可以更换数据目录，但必须明确风险。

流程：

```text
选择新目录
  ↓
检查目录
  ↓
选择迁移方式
  ↓
复制数据或仅切换
  ↓
校验
  ↓
重启应用
```

迁移方式：

- 复制当前数据到新目录
- 使用已有数据目录
- 创建空数据目录

## 9. 删除应用和删除数据

卸载或删除应用不应该默认删除用户数据。

如果用户主动清除数据，必须二次确认。

清除范围：

- 清除应用配置
- 清除源文件
- 清除记忆和索引
- 清除全部数据

默认不清除源文件。

## 10. V1 必须实现

V1 必须有：

- 设置页展示数据目录。
- 打开数据目录。
- 应用版本和数据版本记录。
- 启动时 schema 检查。
- 基础迁移历史记录。
- 迁移前备份数据库和配置。
- 升级失败提示。

V1 可以后置：

- 完整一键回滚。
- 增量备份。
- 云备份。
- 跨设备同步。

## 11. 验收场景

### 场景 1：升级后数据仍在

1. 用户在旧版本导入文件并生成记忆。
2. 用户升级应用。
3. 新版本启动并完成 schema 检查。
4. 用户仍能看到源资料、记忆和图谱节点。

### 场景 2：用户查看数据目录

1. 用户进入设置。
2. 用户看到当前数据目录。
3. 用户点击打开数据目录。
4. 系统打开本地文件夹。

### 场景 3：迁移失败不丢源文件

1. 新版本迁移过程中失败。
2. 系统提示失败。
3. 用户可以查看错误。
4. 源文件仍然存在。
5. 用户可以恢复备份或重试。
